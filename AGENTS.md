# AGENTS.md — mist-schema

## This repo is the canonical schema source

`mist-schema` is the single source of truth for the Mist shared database schema.
**Make all schema changes here.** Do not make schema changes in `mist-web`, `mist-stripe`, or `mist-setup`.

Consumer repos vendor this schema under `third_party/mist-schema/` and their
pre-commit hooks auto-sync latest `main` before running generation or validation.

---

## How to update the schema

### 1. Edit the canonical schema

Edit `db/schema.sql` to reflect the desired end state. This file is the schema
authority — Atlas derives everything else from it.

### 2. Load test database credentials

```sh
set -a
. ./.env.local
set +a
```

`TURSO_DB_URL` and `TURSO_TOKEN` must be set. Credentials are stored in Bitwarden
under the note named exactly `mist-schema secret note`. Never commit, print, or
copy real credential values into this repository.

### 3. Generate an Atlas migration

```sh
atlas migrate diff <change-name> --env turso_test
```

A new file is created in `db/migrations/`. Review the generated SQL before continuing.

### 4. Validate migrations

```sh
atlas migrate validate --env turso_test
```

### 5. Dry-run schema apply against the test database

```sh
atlas schema apply --env turso_test --dry-run
```

This confirms the canonical schema matches the migration history without writing
any changes to the test database.

### 6. Commit and push to main

Stage and commit `db/schema.sql`, the new file in `db/migrations/`, and
`db/migrations/atlas.sum` together. Push to `main`.

```sh
git add db/schema.sql db/migrations/
git commit -m "schema: <description of change>"
git push origin main
```

Tag a release after merge so consumer repos can pin an exact version:

```sh
git tag v<MAJOR>.<MINOR>.<PATCH>
git push origin v<MAJOR>.<MINOR>.<PATCH>
```

## Do not edit generated or vendor output directly

- `db/migrations/*.sql` and `db/migrations/atlas.sum` are managed by Atlas.
  Regenerate them via `atlas migrate diff`; do not hand-edit them.
- `third_party/mist-schema/` in consumer repos is vendored output.
  Update the canonical source here and let consumer pre-commit hooks sync it.

---

## Consumer repos and pre-commit hook behaviour

`mist-web`, `mist-stripe`, and `mist-setup` each vendor this repo under
`third_party/mist-schema/`. Their pre-commit hooks auto-sync the latest `main`
of `mist-schema` before running `sqlc generate` or schema validation.

**Normal workflow:** after schema changes land on `main` here, the next commit in
any consumer repo picks up the latest schema automatically through the hook —
no manual action required.

**If a hook reports schema or generated-code changes:** the hook has detected that
the synced schema or generated output is stale. Stage the reported files and
retry the commit:

```sh
git add third_party/mist-schema/ <any generated files listed by the hook>
git commit   # retry
```

Do not bypass the hook (`--no-verify`). Fix the root cause instead.

---

## Merge discipline

No consumer PR that depends on a schema change may merge until:

1. The corresponding `mist-schema` release tag exists.
2. The consumer repo is explicitly pinned to that release.
3. Consumer CI passes against the pinned schema.

See `docs/schema-policy.md` for the full policy.

---

## What to avoid

- **Do not** make `mist-setup`, `mist-web`, or `mist-stripe` the schema authority.
- **Do not** edit `db/migrations/` files by hand.
- **Do not** point Atlas at production, staging, or any real customer database
  from this repository — the `turso_test` environment is for the dedicated test
  database only.
- **Do not** commit `.env.local` or any file containing real credential values.
